# UNSYS Staking Program

A Solana on-chain program built with Anchor for the Uncertain Systems (UNSYS) ecosystem. It provides dividend staking, tiered partnership referrals, data provider registration, and legacy OMEGA holder migration.

**Program ID:** `8fQT7WjAw2BLYJcbTPYxLciPmUgh5GS4Jj2Vo1uhoK2q`

## Architecture

### On-Chain Accounts

| Account | PDA Seed | Description |
|---|---|---|
| `GlobalConfig` | `"global_config_v3"` | Stores mints, vaults, admin, pending admin, dividend epoch, snapshot pool, pause state, legacy holder count |
| `DividendStake` | `"dividend_stake" + user_pubkey` | Per-user dividend stake with lock period, share multiplier, and epoch tracking |
| `PartnershipStake` | `"partnership_stake" + user_pubkey` | Per-user partnership stake with tier (1/2/3), optional referrer, and per-partner referral balance |
| `DataProviderStake` | `"data_provider_stake" + user_pubkey` | Per-user data provider registration (requires admin activation) |
| `LegacyOmegaStake` | `"legacy_omega" + holder_pubkey` | Admin-registered legacy OMEGA holder with assigned tier |

### Token Types

- **UNSYS** -- Utility/governance token used for staking
- **OMEGA** -- Legacy token for migration benefits
- **USDC** -- Revenue distribution token

### Revenue Distribution Model

**Dividends**: Admin deposits USDC via `deposit_revenue`. 100% goes to the dividend pool. Each deposit increments an epoch. Stakers claim their proportional share once per epoch based on `user_shares / total_shares`. Unclaimed funds roll forward into the next epoch.

**Referrals**: An off-chain tool tracks revenue generated by each partner's referred users and calculates the partner's share based on their tier (10%/30%/50%). Admin deposits the calculated amount per-partner via `deposit_referral_revenue`. Partners claim their full balance anytime.

## Use Cases

### 1. Earning Passive Dividends (Any User)

A user locks UNSYS tokens for a fixed period to earn a proportional share of all USDC revenue deposited by the admin.

**Flow**: `stake_dividends` → (admin calls `deposit_revenue`) → `claim_dividends` → (wait for lock to expire) → `unstake_dividends` → `close_dividend_stake`

- Choose 3/6/12 month lock for 1.1x/1.25x/1.5x share multiplier
- Claim once per epoch (each admin deposit creates a new epoch)
- Longer lock = higher share weight = larger cut of the pool
- After unstaking, close the PDA to reclaim rent, then re-stake if desired

### 2. Becoming a Referral Partner

A user stakes a minimum of 1M UNSYS to become a partner and earn referral revenue from users they bring to the platform.

**Flow**: `stake_partnership` → (refer users) → (admin calls `deposit_referral_revenue`) → `claim_referral_share` → (repeat claims as revenue accumulates)

- Stake amount determines tier and off-chain revenue share:
  - **Tier 1** (1M+): 10% of revenue from referred users
  - **Tier 2** (2M+): 30%
  - **Tier 3** (5M+): 50%
- An off-chain tool tracks which users each partner referred and how much revenue they generated
- Admin deposits the calculated USDC amount per-partner via `deposit_referral_revenue`
- Partner claims their full balance anytime — no epoch restrictions
- To unstake, must first claim any remaining referral balance
- To change tier, fully unstake → close PDA → re-stake at new amount

### 3. Legacy OMEGA Holder Migration

Past OMEGA token holders receive free benefits without locking any tokens — admin registers them and assigns a tier.

**Flow**: (Admin) `register_legacy_holder(wallet, tier)` → (Legacy user) `enable_legacy_benefits` → `claim_dividends` + `claim_referral_share`

- Admin registers the holder's wallet with a tier (1/2/3) — holder doesn't need to sign
- Holder calls `enable_legacy_benefits` to activate both:
  - 10B virtual dividend shares (earns from every revenue deposit)
  - Partnership at the admin-assigned tier (earns referral revenue)
- Admin can deposit referral revenue for legacy partners the same way as regular partners
- Admin can revoke partnership tier via `revoke_legacy_partnership` (dividend shares remain)

### 4. Data Provider Staking

Users stake 5M+ UNSYS to register as data providers. Admin validates them before they become active.

**Flow**: `stake_data_provider` → (admin) `validate_data_provider` → ... → (admin) `deactivate_data_provider` → `unstake_data_provider` → `close_data_provider_stake`

- Must stake at least 5M UNSYS
- Starts inactive — admin must call `validate_data_provider` to activate
- Cannot unstake while active — admin must deactivate first
- After unstaking, close PDA to reclaim rent

### 5. Emergency Admin Operations

The admin can pause all user-facing operations in case of a vulnerability or incident.

**Flow**: `pause` → (investigate/fix) → `unpause`

- Pausing blocks: staking, claiming, deposits, legacy activation
- Pausing does NOT block: admin transfer, account closing, deactivation
- Admin transfer is a two-step process: `propose_admin_transfer` → `accept_admin_transfer` (or `cancel_admin_transfer`)

### 6. Account Lifecycle (Rent Reclamation)

After unstaking, users can close their PDA accounts to reclaim the rent lamports they paid when the account was created.

**Flow**: (unstake fully) → `close_*_stake` → (PDA deleted, rent returned) → (re-stake creates fresh PDA)

- Each close instruction verifies the account is fully zeroed before allowing closure
- Partnership close also requires `referral_balance == 0`
- After closing, the user can create a fresh stake by calling the corresponding stake instruction again

## Security Features

- **Snapshot-based dividend claims**: Revenue is snapshotted at deposit time. Claims calculate from the snapshot, not the live vault balance, eliminating early-claimer advantage.
- **Per-partner referral balances**: Each partner has a dedicated `referral_balance` funded by admin. No global pool sharing or epoch gating.
- **`init` instead of `init_if_needed`**: User stake PDAs use `init` to prevent re-initialization attacks. Defense-in-depth `is_initialized` flags on all stake accounts.
- **Emergency pause/unpause**: Admin can halt all user-facing operations instantly.
- **Two-step admin transfer**: `propose` + `accept` pattern prevents accidental admin loss.
- **Checked arithmetic everywhere**: All operations use `checked_add`/`checked_sub`/`checked_mul`/`checked_div` with `MathOverflow` errors.
- **Tiered partnership minimums**: 1M UNSYS minimum prevents spam registrations.
- **Full unstake only for partnerships**: Prevents partial-unstake tier manipulation. Must claim referral balance first.
- **Legacy holder cap**: Maximum 500 legacy registrations to prevent unbounded dividend dilution.
- **Vault mint and authority validation**: `initialize` verifies vault mints match declared mints and vault authorities are the GlobalConfig PDA.
- **ATA mint validation**: All claim ATAs verify correct mint. All staking ATAs verify correct mint.
- **PDA-verified accounts**: Every instruction verifies accounts via seeds + bump.
- **Signer requirements**: All mutations require owner signature + PDA constraint.
- **Account close guards**: Close instructions enforce fully-zeroed state before allowing rent reclamation.

## Build & Test

### Prerequisites

- [Rust](https://rustup.rs/) (toolchain 1.85.0 pinned via `rust-toolchain.toml`)
- [Solana CLI](https://docs.solanalabs.com/cli/install) v1.18+
- [Anchor CLI](https://www.anchor-lang.com/docs/installation) v0.30.1
- Node.js 18+ and Yarn

### Setup

```bash
yarn install
```

### Build

```bash
anchor build --no-idl
```

Pre-built IDL and TypeScript types are in the `idl/` directory.

### Test

```bash
anchor test --skip-build
```

macOS fix for `._genesis.bin` error:

```bash
rm -rf .anchor/test-ledger test-ledger
COPYFILE_DISABLE=1 anchor test --skip-build
```
